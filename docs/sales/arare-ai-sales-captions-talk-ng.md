## 2. 初回商談トーク台本

### 導入

「店舗運営で、電話の取りこぼしや予約確認の手間はどのくらい発生していますか？」

「ARARE AIは、AIが予約の入口対応を行い、店舗側が確認して確定する予約受付システムです。」

### 課題整理

「接客中に電話が鳴っても出られない、営業時間外に問い合わせが来る、LINEや電話の予約がバラバラになる。こうした予約まわりの負担を減らすことが目的です。」

### 提案

「AIが電話やWebで予約希望を受け、日時・コース・担当希望などを整理します。その後、店舗側のダッシュボードで内容を確認して確定できます。」

### 安全性の説明

「AIが勝手に全部確定するのではなく、店舗が確認して確定する流れなので、現場判断を残したまま効率化できます。」

### 締め

「まずはデモ画面で、電話AI受付から予約確定、通知履歴までの流れを見ていただくのが一番分かりやすいです。」

## 3. デモ配信トーク台本

### 1. 全体ハブ

「こちらが全体ハブです。店舗、セラピスト、顧客、Web予約チャット、電話AIの役割ごとに画面が分かれています。」

### 2. 店舗ダッシュボード

「こちらが店舗スタッフが毎日見る画面です。今日の予約、空室、出勤者、AIが受けた問い合わせがまとまっています。」

### 3. 電話AIログ

「電話AIが受けた内容はここに残ります。通話時間、予約ID、会話要約、予約に紐づいたかどうかを確認できます。」

### 4. 予約作成/編集

「こちらで店舗側が予約を作成・編集できます。日時、コース、セラピスト、部屋、状態を管理でき、空き確認してから保存できます。」

### 5. 通知履歴

「予約確定後の通知は、送信済み・未送信・失敗で確認できます。通知本文も残るので、あとから確認できます。」

### まとめ

「AIが入口対応、店舗が確認、確定後に通知。この流れで、取りこぼしと確認漏れを減らす設計です。」

## 4. 導入提案トーク台本

### 店舗情報ヒアリング

「導入する場合は、店舗名、営業時間、コース、セラピスト、部屋数、予約ルールを登録します。」

### 電話AIの説明

「電話AIを使う場合は、AI受付用の電話番号設定が必要です。既存の店舗番号とのルーティングも運用に合わせて設計します。」

### 通知の説明

「予約確定後の顧客通知やセラピスト通知も設定できます。通知履歴に証跡が残るため、送ったかどうかを確認できます。」

### 導入後運用

「日々の運用では、まず店舗ダッシュボードを見ます。要対応があれば電話ログや通知履歴を確認し、予約は予約一覧で管理します。」

### 次アクション

「次に必要なのは、店舗情報と予約ルールの整理です。そこが揃えば、デモ店舗から実店舗向けの設定に進めます。」

## 5. NG表現リスト

| 言ってはいけない表現 | 理由 | 代わりに使う表現 |
|---|---|---|
| AIが全部自動で予約確定します | 誤確定リスクがあり、現状説明として強すぎる | AIが一次受付し、店舗側が確認して確定できます |
| 電話を100%取りこぼしません | 通信環境、設定、外部サービスに依存する | 取りこぼし削減を目指せます |
| SMSは必ず届きます | 端末、キャリア、Twilio状態に依存する | 送信ログと失敗ログを確認できます |
| LINEもすぐ使えます | 店舗ごとのLINE公式設定が必要 | LINE連携は店舗ごとの設定後に利用します |
| 売上が必ず上がります | 成果保証になる | 予約導線を増やし、機会損失の削減を狙えます |
| AIが店舗の代わりに全部判断します | 店舗確認前提の設計と矛盾する | AIが情報整理し、店舗判断をサポートします |
| どんな質問にも答えられます | 禁止回答や未登録情報がある | 登録済み情報と店舗ルールに沿って案内します |
| グレーな要望もAIが案内できます | 法務/運用リスクが高い | NG回答や店舗確認に回す設計です |
| 初期設定なしですぐ本番運用できます | 店舗情報・電話・通知設定が必要 | 店舗情報と連携設定を入れてから本番運用します |
| 顧客情報は絶対漏れません | 絶対表現は避ける | 認証と権限管理で保護する設計です |

## 6. 営業で強調する安全な表現

- AIが予約受付の一次対応を行います
- 店舗側が確認して確定できます
- 通話ログと通知ログが残ります
- 予約、出勤、部屋、通知を一つの画面で確認できます
- 営業時間外や接客中の問い合わせ対応を補助します
- 店舗ルールを登録することで、AI回答を整えやすくなります
- 完全自動化ではなく、現場確認を残した実用型です

## 7. 資料化前の画像加工チェック

- 電話番号を伏せ字にする
- 顧客名をデモ名にする、または伏せ字にする
- Provider ID、通知ID、予約IDは必要に応じて一部伏せる
- Webhook/Relay URLは営業先に不要なら隠す
- URLバーやアカウント情報が不要ならトリミングする
- スマホの通知、電池、通信情報が気になる場合はトリミングする
