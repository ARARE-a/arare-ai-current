# 最終品質保証ルール

この文書は、AI予約受付MVPを本番提出する前に必ず満たす品質基準を固定するための運用メモです。

## 目的

電話AIを単に動かすことではなく、人間の電話受付と同等以上の品質で、予約取得、SMS送信、管理画面反映まで完了できる状態にする。

## 担当

| 担当 | 役割 | 主な責任 |
| --- | --- | --- |
| A | 電話AI状態管理責任者 | 会話状態、前回日時、保留状態、同一返答回数の管理 |
| B | 日時パース責任者 | 日付、時間、相対日付、曖昧表現、確認質問の制御 |
| C | NLU・意図判定責任者 | キャンセル、紹介、名前確認、おすすめ、空き確認、誤抽出防止 |
| D | 予約エンジン責任者 | 空き確認、予約バリデーション、予約確定、重複防止 |
| E | SMS・通知責任者 | SMS本文生成、送信、失敗保存、通知状態管理 |
| F | 管理画面責任者 | 今日の対応、未来予約、通知失敗、SMS送信状況の分離表示 |
| G | QA責任者 | 自動回帰テスト、失敗分類、合格判定 |
| H | 実電話検証責任者 | 本番電話テスト、call_sid記録、DB/画面/SMS照合 |

## 必須ルール

- 修正者本人が合格判定してはいけない。
- `確認済み`、`未確認`、`推測`、`実装上の判断`、`要ユーザー対応`を必ず分けて報告する。
- 本番未確認のものを本番確認済みと言ってはいけない。
- 外部サービス側の未反映をコード実装だけで解決済みにしてはいけない。
- 1件でも提出条件が未達なら提出可能と言ってはいけない。

## 状態管理の必須項目

```text
requested_date
requested_time
requested_datetime
last_requested_date
last_requested_time
last_requested_datetime
pending_time_confirmation
pending_date_confirmation
selected_therapist
suggested_therapist
customer_name
customer_phone
availability_search_mode
response_repeat_count
```

## 禁止事項

```text
AI提案 = ユーザー指名
前回日時の残留
低信頼データ確定
10時を確認なしで22時確定
2日後を前回日付のまま維持
かない/かなり/空いてる人かないをkanaとして保存
復唱後に空きなしを通知
同じ満枠返答を3回以上繰り返す
```

## 提出可能条件

以下がすべて確認済みになるまで提出不可。

```text
実電話200件以上完了
自動回帰1000件PASS
CSV5000件以上
SMS成功率99%以上
SMS文字化け0件
日時誤認識0件
セラピスト誤認識0件
顧客名誤認識0件
状態破綻0件
同一返答ループ0件
予約確定後エラー0件
DB証跡提出
call_sid提出
Railway本番確認
Twilio本番確認
QA承認
```

## 進行順

```text
担当A/B/C/D/E/Fが修正
↓
担当Gが自動回帰と差分確認
↓
Railway本番反映
↓
担当Hが本番電話検証
↓
DB、SMS、管理画面を照合
↓
全条件達成時のみ提出
```
